# 需求感知層(demand-sensing layer)

> 一套引擎,兩個需求標的:**商品銷售需求** + **維修料件需求**。
> 原則:**熱度只提早示警;實際銷售 / 送修負責驗證與授權。POS 是帳本,指令助理是介面。**
> 配合 [assistant-command-layer.md](assistant-command-layer.md)、[data-model.md](data-model.md) 看。

## 0. 為什麼是「一套引擎、兩個標的」

商品需求與維修需求形狀完全一樣:

```
訊號 → 熱度分數(相對成長) → 轉換率(學來的) → 預估需求 → 受限建議 → 警示 → 人確認 → 下單
```

差別只在**標的**與**資料鏈**:
- 商品:機型 / SKU。
- 維修:機型 → 症狀 → 維修項目 → 料件。

所以**共用同一組訊號 / 警示資料表與服務,不做兩套系統**。`apps.signals` 疊在既有帳本之上,不改動帳本。

## 1. 四層決策(兩個標的共用)

```
市場熱度訊號
      ↓
實體銷售 / 送修驗證
      ↓
生命週期 + 跌價 / 庫齡風險
      ↓
安全存量 / 調貨 / 叫貨建議
```

**貫穿全部的授權閘**:外部熱度只准「抬警示」;真正上修數量,要等**內部領先訊號**跟上才放行。
- 商品:詢問 / 預購 / `velocity`。
- 維修:報價紀錄 / 工單 / 缺料流失案件。

> 熱度是**領先指標但有 lag**。很多產品發表會後搜尋暴增卻沒人買——直接依熱度叫貨就堆出一座漂亮的庫存山。這道閘就是防那座山。

## 2. 熱度分數(看相對成長,不是絕對值)

第一版簡單權重(商品):

| 訊號 | 權重 |
|---|---|
| Google 搜尋成長率 | 30% |
| 自家詢問量成長率 | 25% |
| 預購 / 訂金 | 20% |
| YouTube 討論增速 | 10% |
| 供應商缺貨程度 | 10% |
| 促銷活動 | 5% |

但真正要看的是:近 7 天 vs 前 7 天成長多少、是否**連續**上升、上升後**幾天**反映到銷售(lag)、是否只集中在發表會當天、**台灣全國熱度是否符合本門市區域**。

**轉換率要學,不能直接套**:搜尋 +50% 歷史上可能只帶來門市 +12% 成交。系統學的是那個 12%。
冷啟動用階梯 fallback:`SKU → 機型 → 類別 → 全域`,新機沒歷史先借前代機種。

## 3. 生命週期權重 + 跌價 cap(商品)

| 階段 | 熱度的角色 |
|---|---|
| 上市前 | 搜尋 / 詢問 / 預購 / 前代機種為主 |
| 首發期 | 熱度與實際銷售各占較高比重 |
| 成長期 | 用熱度提早偵測銷量加速 |
| 成熟期 | 實際銷售為主,熱度只做修正 |
| 衰退期 | 即使仍有熱度,也受跌價風險限制 |
| 停產期 | 除非有客訂,否則不因短期熱度恢復補貨 |

硬性上限:`建議量 = min(熱度修正後預測, 跌價風險上限(生命週期, 上市天數))`。

**型號 / 顏色 / 容量分開預測**:外部熱度預測**機型**;POS / 預購決定**容量**比例;門市歷史決定**顏色**比例。
**資料品質命門**:缺貨時要記錄客戶原本想要的容量 / 顏色,否則替代購買會把真實需求洗掉。

**接點(不重造)**:`catalog.Product` 已有 `velocity_ewma / velocity_recent_14d / velocity_baseline_90d / trend_ratio / dynamic_safety_stock / lifecycle_status`(由 `compute_dynamic_stock` 算)。熱度層產出一個**修正係數**餵進 `dynamic_safety_stock`,**不取代**它。

## 4. 維修需求 track(第二個標的)

維修熱度可能比商品熱度更接近實際消費——搜「某型號換螢幕價格」的人通常已有明確需求。

**資料鏈**(接既有 `apps.repairs` + `catalog.PartTemplate`):

```
手機型號 → 顧客症狀(Symptom) → 維修項目(Resolution) → 使用料件(SparePart / 零件倉 SKU)
```

| 搜尋 / 症狀 | 維修項目 | 可能料件 |
|---|---|---|
| 電池耗電、膨脹 | 電池更換 | 電池、背膠 |
| 螢幕破裂、觸控失效 | 螢幕維修 | 螢幕總成、防水膠 |
| 無法充電、接觸不良 | 充電檢測 | 尾插、排線 |
| 相機模糊、無法對焦 | 相機維修 | 相機模組 |
| 喇叭無聲、通話小聲 | 音訊維修 | 聽筒、揚聲器、防塵網 |

**同一症狀不一定同一料件 → 初期只給備料「建議」,不自動採購。**

**維修需求公式**:

```
預估料件需求 = 機型市場保有量 × 裝置年齡故障機率 × 維修搜尋熱度 × 送修轉換率 × 該店市占 / 接單能力
```

關鍵洞見:某舊機型**購買熱度已下降,但保有量大、進入電池老化期** → 電池維修需求反而持續上升。所以維修 track 的核心不是購買熱度,是**裝置年齡故障曲線 × 保有量**。

**搜尋意圖分級**(高意圖對備料參考價值明顯高於 DIY / 討論):
- 高:型號＋換螢幕價格 / 電池更換 / 維修費用 / 無法充電維修 / 附近維修店
- 中:型號＋故障 / 耗電 / 觸控失效 / 相機異常
- 低:型號＋拆機 / DIY / 故障原因

**料件不能同一套備貨邏輯**:

| 料件特性 | 策略 |
|---|---|
| 低成本、跨機型共用 | 較高安全存量 |
| 高成本、機型專用 | 少量備貨或接單後採購 |
| 交期長、缺貨損失高 | 提高安全存量 |
| 價格快速下跌 | 限制庫存金額 |
| 電池等有保存期限 | 批次 + 庫齡 + FIFO |
| 品質不穩定 | 不良率進採購量 + 供應商評分 |

**維修叫貨公式**:

```
建議採購量 = 預估交期內維修件數 + 動態安全存量 − 現有良品料件 − 在途料件
```
同時設「最大庫存金額」與「最長庫存天數」,避免維修熱度退燒後留一堆專用料件。

**接點**:`repairs`(RepairOrder / Symptom / Resolution / SparePart 工單與實際用料)+ `catalog.PartTemplate / PartTemplateItem`(已有 `default_cost / default_safety_stock / shared_across_models`)+ 零件倉(`Product.warehouse_type=parts` 的 `StockBalance`)。

## 5. 訊號來源(內部優先)

**內部最重要**(商品):POS 實際銷量、客戶詢價次數、門市 / 官網搜尋、預購訂金、缺貨後未成交需求、不同容量顏色詢問量。
**內部最重要**(維修):維修報價、實際工單、缺料流失案件、每項維修實際用料、返修 / 料件不良率、維修毛利與完修時間。
**外部**(提早示警,兩者共用):Google Trends、Google Ads 關鍵字量、YouTube 影片 / 觀看增速、論壇 / 社群故障討論、原廠公告、機型上市時間與保有量、供應商交期 / 成本 / 缺貨。

> 外部 API 現況(2026-07):Google Trends 官方 API 2025 推出,至今仍**限量 alpha、未 GA**——初期**每週匯入 CSV** 即可,不必等正式 API。Google Ads API 可取歷史關鍵字量(中長期需求),YouTube API 可取影片統計。社群 / 電商排名務必**合法 / 合 ToS** 取得。

## 6. 授權閘 + 回測門檻(安全的核心,兩者共用)

1. **熱度只示警,內部訊號授權。** 外部熱度單獨永遠不自動加叫貨量;要內部領先訊號同步上升才放行。
2. **回測過關才進自動補貨。** 證明加入熱度後比純銷售資料**降低 10–15% 預測誤差**,才放進公式。
   量:預測誤差(WAPE)+ 生意指標——商品:缺貨率↓ / 呆滯跌價庫存↓;維修:缺料流失↓ / 當日完修率↑ / 料件周轉天數 / 呆滯料件金額 / 供應商不良率。

## 7. 資料模型(generic,商品 / 維修共用)

新 app `apps.signals`,一律 `TenantOwnedModel`:

| Model | 用途 | 關鍵欄位 |
|---|---|---|
| `SubjectAlias` | 別名庫:把「17 Pro / i17P / iPhone 17 Pro」對到同一標的 | `alias`、`subject_key`、`product`(nullable)、`kind`(product_sales / repair_part) |
| `MarketSignal` | 一個訊號時間點 | `source`、`subject_key`、`period_date`、`value`、`meta` |
| `ConversionModel` | 學到的熱度→成交轉換率 + lag(分層) | `scope`(sku/model/category/global)、`subject_key`、`conversion`、`lag_days` |
| `DemandAlert` | 需求上升 / 下降 / 備料預警 | `kind`、`subject_key`、`product`、`direction`、`heat_growth`、`internal_growth`、`authorized`、`status` |

`subject_key` 是共用鍵:商品用機型 key(例 `apple-iphone-15-pro`),維修用 `機型+症狀`(例 `apple-iphone-12__battery`)。同一組表兩個標的都吃。

## 8. 與指令助理合流

`apps.signals` 只產「需求上升 / 下降 / 備料預警」。要下單時,交給**指令助理**:

```
DemandAlert(上升,已授權) → 指令助理:「這些型號建議上修、這些退燒建議收手」
        → 你確認 → 生一張 draft 叫貨單(商品=create_purchase_order;料件=同 intent 指向零件倉 SKU)
        → commit → 帳本記準
```

熱度層負責「**早知道**」,指令層負責「**一句話下單**」,帳本負責「**記準**」。三層合起來就是產品。

## 9. 導入計畫(各 track 8 週,先示警不叫貨)

**商品**:①② 週建 SKU 別名庫 → ③④ 週每週匯 Trends CSV 對照 POS → ⑤⑥ 週找 lag 與轉換率 → ⑦⑧ 週只顯示「需求上升 / 下降警示」→ 回測過門檻才進自動補貨。

**維修 MVP**(先別監控所有機型 / 故障):
1. 匯入過去 6–12 個月維修工單。
2. 找出占營收 80% 的**前 20 組「機型—維修項目—料件」**。
3. 建相關搜尋字詞群(依意圖分級)。
4. 每週取熱度,與實際維修件數對照,找 lead time。
5. 先給備料預警,累積 8–12 週再談自動採購。

## 10. 護城河

多家維修 / 通訊業者使用後,在**不含客戶個資**前提下,匿名彙整「地區 × 機型 × 症狀 × 週次」→ 一套別人很難取得的**維修故障趨勢資料庫**。這比「自動化便利」難複製得多,可能是最強的商業護城河。

> 一句話定位:你賣的不是自動登入 POS,是一套**比店員更早發現要熱賣、也更早知道要退燒**,連**下一批會壞什麼、該先備哪些料**都算得出來的通訊業庫存決策系統。

## 11. 已知限制 / 待辦

- 冷啟動(新機無史)→ 借前代 + 類別先驗。
- 區域偏差 → 學本店對全國趨勢的相關係數再加權。
- lag 估計 / 同症狀多料件 / 資料品質(缺貨規格捕捉)→ 都要當一等公民。
- 自動採購前務必守「授權閘 + 回測門檻」兩關。
