# 新經銷商上線 — 資料索取清單

> 用途:給準備要從舊 POS 系統(歐睿、福斯、其他廠牌)轉到 MP POS 的新經銷商,
> 用來向對方索取資料時的清單。內容直接可轉發給經銷商或對方原 POS 廠商。

---

## 一、上線流程概述

```
1. 經銷商從現有 POS 匯出以下 5 類資料(Excel 或 CSV)
       ↓
2. 將檔案交給平台管理者(=本系統供應商)
       ↓
3. 平台管理者核對格式 → 跑 dry-run 預覽 → 經銷商確認筆數正確
       ↓
4. 平台管理者正式寫入 MP POS 資料庫
       ↓
5. 經銷商登入新系統驗收
```

**整體所需時間**:資料正常的話從交檔到上線約 1–3 個工作天。
**經銷商不需要寫程式或操作技術細節**,只要把舊系統的資料匯出來即可。

---

## 二、必要資料(必交)

以下 5 類為**完整轉移所需的最低項目**。若舊 POS 沒有某些欄位也沒關係,平台管理者會協助評估。

### 1. 商品主檔(品項清單)

每一個販售的品項一行。

| 欄位 | 必要 | 範例 | 說明 |
|---|---|---|---|
| 品號 / SKU | 必要 | PH-000123 | 品項在舊系統的唯一代碼 |
| 品名 | 必要 | iPhone 15 Pro 128G 黑色 | 完整品名 |
| 規格 | 選 | 128G / 黑色 | 若品名已包含可省略 |
| 類別 | 必要 | 手機 / 配件 / 中古機 | 大分類即可 |
| 條碼 | 選 | 4710123456789 | 原廠條碼,有就帶 |
| 建議零售價 | 選 | 32900 | 含稅單價,沒有可留空 |

**檔名建議**:`商品主檔.xlsx`

---

### 2. 庫存(序號型商品 — 手機 / 中古機 / 平板)

需要逐支追蹤 IMEI / 序號 的商品,每一支一行。

| 欄位 | 必要 | 範例 | 說明 |
|---|---|---|---|
| 品號 | 必要 | PH-000123 | 對應上面商品主檔的品號 |
| IMEI / 序號 | 必要 | 354123456789012 | 該機的唯一序號 |
| 倉別 | 必要 | 南大店 / 北倉 / 中央倉 | 該機現在所在門市 |
| 中古機成色 | 選 | S / A / B / C / D | 中古機才需要 |
| 中古機售價 | 選 | 9900 | 中古機每支獨立定價 |
| 電池健康度 | 選 | 95 | 中古機才需要 |
| 進貨成本 | 選 | 28000 | 該機原始進貨成本(有最好) |

**檔名建議**:`庫存_序號.xlsx`

## 四、格式與檔案規範

| 項目 | 規範 |
|---|---|
| 檔案格式 | `.xlsx` / `.xls` / `.csv` 都可以,**推薦 .xlsx** |
| 文字編碼 | CSV 請存成 **UTF-8 with BOM**(用 Excel 另存新檔時選擇) |
| 欄位順序 | 不一定要照上述順序,但**標題列**(第一行)請寫清楚欄位名稱 |
| 空值 | 留空格即可,不要填 `N/A`、`-` 或 `無` |
| 日期格式 | `YYYY-MM-DD`(例 2024-08-12)。若舊系統匯出是其他格式請註明 |

## 五、常見問題

**Q1. 我的舊 POS 沒辦法匯出某個欄位,怎麼辦?**
A:先把有的欄位匯出來,沒有的欄位留空即可。平台管理者收到後會評估影響。

**Q2. 我的商品品號和新系統不一樣,會自動對應嗎?**
A:**不會**,新系統會沿用您**舊系統的品號**作為 SKU(除非特別約定)。
這樣優點是「對得上歷史紀錄」,缺點是後續新進商品才會用新系統的編號規則。

**Q3. 中古機每一支都不一樣,要怎麼匯?**
A:中古機在新系統是「一支一筆 ProductSerial」,所以**庫存_序號.xlsx** 裡面
每一支中古機都各佔一行,並填上成色 / 售價 / 電池健康度。

**Q4. 倉別怎麼決定?**
A:平台管理者上線前會跟您確認所有門市清單,把每間門市建為一個「倉別」。
資料匯入時,「倉別」這欄填**門市名稱**(例:南大店、北倉)即可。

**Q5. 過去的進貨單 / 調撥單要不要匯?**
A:**不需要**。新系統只保留現在「庫存還在的」品項與「會員消費紀錄」,
過去的單據(進貨單號、調撥單號)不還原成新系統的單,因為意義不大且資料量太大。

**Q6. 我有報價單 / 維修單在舊系統,要不要匯?**
A:目前(2026/05)新系統的維修模組尚未實作,暫時無法匯入。
等該模組上線後若您仍有需求,可另議。

**Q7. 匯入會不會覆蓋我已經在新系統建好的資料?**
A:會。匯入流程預設**先清空再寫入**(僅清空商品 / 序號 / 庫存 / 會員,
不會清付款方式 / 發票設定 / 系統參數)。**故新經銷商務必在「未開始操作新系統」
時匯入,以免覆蓋自己手動建好的資料**。

## 六、平台管理者內部 SOP(僅平台方參考)

> 此節經銷商可忽略。

1. 收到 Excel → 放到 `/path/to/onboarding/{tenant_code}/` 目錄
2. 看 5 個檔案的欄位 → 對照本清單評估有缺哪些
3. 若該 POS 來源已寫過 import command(如 `import_legacy_inventory`) → 直接調整 mapping
   若是新 POS 來源 → 在對應 app 的 `management/commands/` 新建 `import_{vendor}_*.py`,
   仿照既有檔案結構(default dry-run、`--confirm` 才寫入、寫入前包 transaction)
4. 跑 dry-run:`python manage.py import_xxx --xls /path/檔案.xlsx`
   檢查印出來的「會匯入 N 筆商品 / M 筆序號 / K 筆會員」對不對
5. 截圖 / 數字回報給經銷商確認 → 經銷商 OK 後 → 加 `--confirm` 正式寫入
6. 寫入完成 → 用 platform admin 帳號登入該 tenant 抽檢 5–10 筆資料 → 通知經銷商驗收
