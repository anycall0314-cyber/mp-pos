## 北極星:拍一張出貨單就入庫

一句話講清楚要蓋到哪:

> **就算遇到沒往來過的新廠商,老闆拍一張他的出貨單,就能把貨入庫。**

這句話拆開來,正好是後面三塊地基疊起來的成果:

- **熟廠商**:拍照 → 系統自動把出貨單品項對到我的商品 → 瞄一眼確認 → 入庫。接近全自動。
- **全新廠商(第一次)**:拍照 → OCR 拆出品項 → 認得的直接帶入、**認不出的進「待確認區」**讓你點兩下對應(或建一筆別名) → 確認 → 入庫。這一次你就「教」會了系統這家廠商的講法,下次同一家就順了。

所以「拍照即入庫」不是一個開關,是**每教一次就更聰明一次**的能力。而且這正是護城河:每對一次品名,跨廠對照庫就厚一層;將來多店共用時,別家教過的廠商,你第一次拍就認得。

這也解釋了為什麼「商品識別 + 待確認區」必須先蓋——**它就是讓「新廠商」這個最難的情況變安全的機制**(認不出就擱著,絕不亂寫正式庫存)。

## 建議的蓋法(照這順序)

### 第 0 階段(進行中):指令台前端

「AI 是操作介面」的第一個看得見成果。新頁 `/assistant`「指令台」:打字開進貨單 → 提案卡 → 確認 → 真的建單。入口放頂列;這一步追問卡只顯示問題、由人改寫重送(互動式選候選補回留到下一步)。小、能 demo、能自己店先用。

### 第 1 階段:商品識別 + 待確認區(§7 最小版) ← 真正的地基

- **蓋什麼**:一層「平台標準商品」＋底下掛「廠商別名／料號」＋條碼/料號唯一約束＋一個**待確認區**(認不出的東西不准動正式庫存,先擱著等人確認)。
- **為什麼排這**:後面每一塊都要先能「把廠商亂七八糟的品名對到我的商品」——OCR 要它、到貨核對要它、熱度對機型要它、補貨建議要乾淨主檔。沒有它,後面全是砂上蓋樓。也是「拍新廠商出貨單就入庫」能安全成立的關鍵。
- **先做規則版**:條碼 → 料號 → 完整屬性精準比 ＋ 信心門檻(98% 自動 / 85–98% 人選候選 / 85% 以下標未知 / 關鍵欄位衝突禁止自動)。AI 語意推測晚點再加。

### 第 2 階段:到貨核對 + 掃描入庫

- **蓋什麼**:出貨單品項對到主檔後,掃實物條碼/IMEI/數量,系統比對「短少、錯貨、重複序號、成本差」,人確認才寫進 POS。
- **為什麼排這**:這是願景第二章講的每天人工痛點(叫貨、對單、入庫),風險低(只是核對、人按確認),做出來馬上有感。
- **依賴**:要 ① 先能認品項。

### 第 2b 階段:進貨單 OCR 辨識(拍照/PDF)

- **蓋什麼**:拍出貨單 → OCR 拆出品名/數量/單價/序號 → 丟給 ① 對應、丟給 ② 核對。連打字都省。
- **這一步做完,「北極星」就成立**:拍照即入庫(熟廠商全自動、新廠商教一次就順)。
- **紀律**:最脆弱的一段,先只吃你最常遇到的 1–2 家供應商格式,再往外擴。

### 第 3 階段:補貨/調貨建議閉環 ← 「代理人」核心

- **蓋什麼**:動態安全存量 ＋ 成熟熱度(先補 ConversionModel 讓它會學轉換率)→ 算出「先調哪些、再叫哪些」的建議 → 推 LINE 或後台給主管核准 → 核准後產草稿叫貨單 → 交指令助理過帳。
- **為什麼排這**:把前面零件串成一個閉環。但 R&D 最重、要累積幾週資料,擺在地基與到貨核對之後。
- **紀律**:先出建議、人核准;回測證明降 10–15% 誤差才准自動。

### 第 4 階段:維修料件引擎(§6)

- **蓋什麼**:匯入維修工單 → 建「機型×症狀×料件」關係 → 用裝置年齡故障曲線＋保有量算料件備貨建議。
- **為什麼排這**:第二條產品線／加購模組,可獨立賣。技術上跟熱度共用同一套骨架(signals 已預留 `repair_part` 標的類型),① ③ 成熟後接它最省力。

### 第 5 階段:供應商串接 / RPA / 採購上限 / 機器人帳號

- **蓋什麼**:真正自動下單到供應商系統,加單次/單日採購上限、獨立機器人帳號、憑證加密。
- **為什麼最後**:風險最高、法遵最重(§9)。按願景第八章原則「官方 API → 匯出介接 → 瀏覽器自動操作」,能用 API 就別 RPA,要一家家談授權,急不得。

### 長期:跨店匿名資料網路(§12)

多家業者使用後,在不含客戶個資前提下匿名彙整「地區×機型×症狀×週次」,形成別人難複製的預測資料網路。沒規模就沒有它,放最遠。

## 跨階段:自然語言下單用哪個模型

第二個北極星:**經銷商直接用中文對 POS 下指令**(白話下單)。這塊在指令助理成熟度上屬於「開自然語言模式」那一步(目前 LLM 預設關閉、只吃半格式化)。

定案方向:

- **AI 只當翻譯,不碰執行。** 模型只把人話翻成固定格式的 Intent,真正寫入一律走既有會驗證的 service。所以模型可以便宜、可以隨時換,翻錯了驗證網照樣擋。
- **正式環境目標用 DeepSeek**(算力便宜、中文解讀強)。這個任務很窄(中文 → 固定 JSON schema)、又有驗證網兜底,正是可以安全用便宜模型的場景。
- **可混用省錢**:先跑免費的 DeterministicParser,看不懂才丟 DeepSeek,把 API 花費壓到最低。

兩個已知坑:

1. **provider 切換還沒真的實作。** `parsers.py` 的 LLMParser 雖有 `ASSISTANT_LLM_PROVIDER` 設定,但 `_complete()` 目前寫死 Anthropic Messages API 格式(`api.anthropic.com/v1/messages`、`x-api-key`、回傳 `content[0].text`)。DeepSeek 走 OpenAI 相容格式(`api.deepseek.com/.../chat/completions`、`Authorization: Bearer`、回傳 `choices[0].message.content`,model 用 `deepseek-chat`),要補一段真的 provider 分支才能換。(dev 預設仍用 `claude-sonnet-4-6`,原本寫死的無效 `claude-sonnet-5` 已修掉。)
2. **資安 / 法遵(§9)。** DeepSeek 是中國供應商。**進貨單**內容(供應商/商品/序號)無客戶個資,送去翻譯疑慮小;但擴到**銷貨**會碰到姓名/電話/身分證/門號——這類指令必須先去識別化(遮罩)再送,或乾脆走本地解析不送外部。

## 核心紀律(每個階段都套)

1. 先調貨,再叫貨。
2. 熱度是領先訊號,不是直接採購依據。
3. 安全存量隨產品生命週期動態變化。
4. 進貨單數量必須與實際收到的核對。
5. 名稱不同不代表商品不同。
6. 無法識別不代表是新品。
7. 高風險操作必須可核准、可追蹤、可撤銷。
8. 寧可暫存等待確認,也不錯誤異動正式庫存。

## 如果只能挑一件事往下走

做完第 0 階段(指令台前端)後,**立刻投第 1 階段(商品識別地基)**。它最不好 demo,卻是唯一「你不做、後面全做不了」的東西,也正是願景裡最像護城河、現在卻最空的那塊。晚做,等於讓後面每一塊都欠它的債——包括你最想要的「拍照即入庫」。
